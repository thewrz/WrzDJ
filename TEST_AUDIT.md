# Test Suite Audit ‚Äî WrzDJ

Generated by Phase 1 discovery. No code has been changed.

## Suite overview

| Package | Files | Test cases | Framework | Coverage gate |
|---------|-------|-----------|-----------|---------------|
| `server/` | 96 | ~2,260 | pytest + TestClient | 85% total |
| `dashboard/` | ~75 | ~972 | vitest + RTL | 70% branch / 65% fn / 78% line |
| `bridge/` | 14 | ~342 | vitest (Node) | 80% all |
| `bridge-app/` | 7 | ~72 | vitest (jsdom) | 55% branch / 70% fn / 75% line |
| `kiosk/wifi-portal/` | 1 | ~40 | pytest (not in CI) | ‚Äî |

**Pyramid shape:** Backend is integration-heavy (TestClient + real SQLite boundary).
Pure unit tests exist for isolated services (camelot, dedup, track_normalizer,
intent_parser, version_filter, priority_scorer, camelot). Frontend is component
+ hook unit tests with no E2E layer. Bridge has solid unit coverage. Bridge-app
is under-tested relative to the complexity it ships to DJs' laptops.

**Observed CI runtime from workflow definition:** All 4 jobs run in parallel on
ubuntu-latest. Backend is the slowest (PostgreSQL service, 2,260 tests, bandit,
pip-audit). No timing data in the workflow definition, but the parallelism is
correct.

## Anti-patterns to remove or rewrite

### 1. `test_reversibility.py` ‚Äî filesystem/docs assertions (DELETE)

**File:** `server/tests/test_reversibility.py` (2 tests)  
**Tests:**
- `test_recovery_doc_exists` ‚Äî `assert doc.exists()`
- `test_recovery_doc_contains_restoration_keywords` ‚Äî checks for substrings in markdown

**Problem (Khorikov properties):**
- *Protection against regressions:* zero. The recovery procedure is only needed if
  a developer actively re-introduces IP identity, not during normal operation.
- *Resistance to refactoring:* fails if the docs file is moved, renamed, or split
  without any behavior change.
- *Maintainability:* confuses future contributors who wonder why a test checks markdown.

These tests pin docs file structure, not application behavior. The right home for
this constraint is a `# NOTE: see docs/RECOVERY-IP-IDENTITY.md` comment in the code
that actually removed the feature, not a test runner.

**Action:** Delete both tests. The real regression protection for the IP-identity
removal is in `test_no_ip_identity.py` and `test_no_ip_logging.py`, which are sound.

### 2. Help-system test files ‚Äî cross-page boilerplate repetition (REWRITE / CONSOLIDATE)

**Files (8 files, ~32 tests):**
- `dashboard/app/admin/ai/__tests__/ai-help.test.tsx`
- `dashboard/app/admin/events/__tests__/events-help.test.tsx`
- `dashboard/app/admin/__tests__/overview-help.test.tsx`
- `dashboard/app/admin/integrations/__tests__/integrations-help.test.tsx`
- `dashboard/app/admin/settings/__tests__/settings-help.test.tsx`
- `dashboard/app/admin/users/__tests__/users-help.test.tsx`
- `dashboard/app/events/__tests__/events-help.test.tsx`
- (more in `events/[code]/`)

Each file contains the same 4-test pattern:
1. `HelpButton renders on X page` ‚Üí `screen.getByRole('button', { name: 'Toggle help mode' })`
2. `HelpSpots exist for [X, Y, Z]` ‚Üí multiple `getByTestId('help-spot-...')` assertions
3. `first visit triggers onboarding after delay`
4. `subsequent visits do not auto-trigger onboarding`

**Problems by test:**

**Tests 1, 3, 4** are testing `HelpContext` / `HelpProvider` behavior, not page-specific
behavior. These three patterns are *identical* across all 8 files (different page
renders, same assertions). `HelpContext.test.tsx` already exists in
`dashboard/lib/help/__tests__/` and should be the home for this. Running the same
context behavior test 8 times provides no additional regression value ‚Äî one more
passing variant of an already-passing test cannot catch a new bug.

**Test 2** (`HelpSpots exist`) is a **change detector**. It asserts on `data-testid`
attribute values (`help-spot-admin-ai-key`, `help-spot-admin-ai-enable`, etc.) which
are internal markup identifiers, not observable user-facing behavior. A developer
renaming a help-spot ID during refactoring fails this test with no user-visible change.

**Action:**
- Delete tests 1, 3, 4 from all 8 help files (already covered by `HelpContext.test.tsx`).
- Delete test 2 entirely. HelpSpot existence is verifiable by exercising the help flow
  in `HelpContext.test.tsx` using a real page render, not by asserting on test-id strings.
- If you want spot-coverage regression, test the rendered *content* shown when a spot
  is activated (the tooltip text), not the presence of the element with a specific ID.
- Needs human judgment: if `HelpContext.test.tsx` does not already cover the
  first-visit/subsequent-visit onboarding cycle end-to-end with timer mocks, add one
  consolidated version there before deleting the scattered copies.

### 3. `collect-api.test.ts` ‚Äî URL/method assertion change detectors (REWRITE)

**File:** `dashboard/lib/__tests__/collect-api.test.ts`  
**Affected tests:** ~8 of the 12 tests in the file; examples:
- `getCollectEvent issues GET /api/public/collect/{code}` ‚Äî half the assertion is
  `expect(fetch).toHaveBeenCalledWith(expect.stringMatching(/...$/), ...)`
- `submitCollectRequest POSTs JSON with credentials` ‚Äî asserts on `method: "POST"` and
  `credentials: "include"`; the return value check is in a *different* test.

**Problem:** `expect(fetch).toHaveBeenCalledWith(url_pattern, options)` pins the *how*
of the HTTP call (URL shape, method string, credentials flag). These fail when the URL
is versioned (`/v2/`), when `credentials` is moved to a base config, or when the client
is refactored to use `axios` ‚Äî all without any change to what data the caller receives.

The tests that *only* check the return value (e.g., `submitCollectRequest returns
is_duplicate flag`, `throws ApiError on 409`) are sound and should be kept as-is.

**Action:**
- For each test that asserts on `fetch` call arguments AND also checks a return value,
  remove the `toHaveBeenCalledWith` assertion and keep the return-value assertion.
- For the few tests where the only assertion is the call pattern (e.g., `issues GET ...`),
  rewrite to assert on the returned data, or delete if the response is already covered
  by an adjacent test.
- Exception: `credentials: "include"` check for public endpoints is security-relevant.
  If it needs to be tested, extract it into one single `credentials-policy.test.ts`
  focused on that contract, not mixed into 8 happy-path tests.

### 4. `api.test.ts` ‚Äî DOM manipulation assertions in CSV export test (REWRITE)

**File:** `dashboard/lib/__tests__/api.test.ts`  
**Tests:**
- `exportPlayHistoryCsv triggers browser download` ‚Üí asserts:
  `createElementSpy.toHaveBeenCalledWith('a')`,
  `mockAnchor.click.toHaveBeenCalled()`,
  `revokeObjectURLSpy.toHaveBeenCalledWith(mockUrl)`

**Problem:** These assert on the DOM-manipulation *mechanism* used to trigger a
browser download (create an `<a>`, set `.href`, call `.click()`). If the same user
observable behavior (file downloads) is achieved via `showSaveFilePicker` or any
other approach, the test breaks ‚Äî pure change detector.

**Action:** Rewrite to assert on observable side effects: the URL hit (`/api/events/{code}/export/play-history/csv`), that the response blob was fetched, and that `URL.createObjectURL` was called (the data preparation step). Delete the `createElement('a')` and `anchor.click()` assertions.

### 5. `test_cors.py` ‚Äî redundant `test_cors_methods_include_put` (DELETE)

**File:** `server/tests/test_cors.py`  
**Test:** `test_cors_methods_include_put` ‚Äî `assert "PUT" in CORS_ALLOW_METHODS`

**Problem:** `test_cors_methods_cover_all_api_routes` already iterates every route
registered on the app and checks each HTTP method is present in `CORS_ALLOW_METHODS`.
If PUT were removed, the broader test would fail. The PUT-specific test adds no new
information and requires updating if an endpoint using PUT is ever removed.

**Action:** Delete `test_cors_methods_include_put`. The regression story it was written
for (PUT missing from non-wildcard origins) is fully covered by the route-scanning test.

---

### 6. `test_event_bus.py` ‚Äî private attribute assertion (REWRITE)

**File:** `server/tests/test_event_bus.py`  
**Test:** `test_unsubscribe_cleans_up_empty_channel` ‚Äî asserts `"EVT001" not in bus._channels`

**Problem:** `_channels` is a private implementation attribute. Asserting on it pins
the internal dict structure. If the bus is refactored to use a `defaultdict` that keeps
empty keys, the test fails despite correct behavior.

**Action:** Replace with a behavior-observable assertion: after unsubscribing, `bus.subscriber_count("EVT001") == 0`. The count already being tested implicitly covers channel cleanup. If internal cleanup is a memory concern, add a comment explaining the expected behavior, not a test on private state.

### 7. `fonts.test.ts` ‚Äî tautological string checks (DELETE or SHRINK)

**File:** `dashboard/lib/__tests__/fonts.test.ts`  
**Questionable tests:**
- `defines a body font family` ‚Äî `toBeDefined()`, `typeof ... === 'string'`, `length > 0`
- `defines a display font family for kiosk headings` ‚Äî same pattern
- `defines a mono font family for numeric badges` ‚Äî same pattern
- `returns body class for body role` ‚Äî `expect(getFontClass('body')).toBe('font-body')`

**Problem (tautological):** The first three tests verify that a string constant is a
non-empty string. They cannot distinguish between `FONT_FAMILIES.body = "Arial"` and
`FONT_FAMILIES.body = "garbage-string"`. The `getFontClass` tests are mirroring the
implementation lookup table.

The one genuinely useful test is `all font families include a fallback` (checks CSS
fallback stacks: sans-serif, monospace). That one should stay.

**Action:** Delete the 3 `isDefined + isString + length > 0` tests. Delete the 3 `getFontClass` returns-X-for-X tests. Keep `all font families include a fallback`. The file would shrink from 35 lines to ~8.

### 8. `theme-vars.test.ts` ‚Äî exact-count change detector (REWRITE)

**File:** `dashboard/lib/__tests__/theme-vars.test.ts`  
**Test:** `THEMES constant contains exactly three themes in order` ‚Äî `expect(THEMES).toEqual(['dark', 'high-contrast', 'daylight'])`

**Problem:** This test fails when a 4th theme is added, a reasonable extension that
has no user-visible regressions for existing themes. It pins the exact array contents
and order, not the invariants of the THEMES system.

**Action:** Replace with property-invariant assertions: (a) THEMES is a non-empty
array, (b) every element in THEMES has a corresponding entry in `getThemeVars`, (c)
the three existing themes are present. An exact-equality on order is unnecessarily rigid.

### 9. Needs human judgment ‚Äî `test_no_ip_storage.py` absence tests

**File:** `server/tests/test_no_ip_storage.py` (5 tests)  
**Tests:** All assert `not hasattr(Model, "attribute")` or that DB columns don't exist.

**Classification unclear:** These are unusual tests that pin the *absence* of attributes.
They're regression guards against re-introducing IP storage ‚Äî the intent is sound (the IP
identity removal has a documented production history). However, they test model structure
rather than behavior. If the attribute name changes (e.g., `ip_fingerprint` instead of
`ip_address`), the test passes but the regression silently re-enters.

**Recommendation:** Keep as-is for now. The model-attribute checks are cheap and the
documented production incident justifies the unusual form. However, a stronger test
would assert behavioral outcomes: two guests from the "same IP" get different profiles
(which `test_no_ip_identity.py` already covers). Consider deleting the model-attribute
assertions after confirming `test_no_ip_identity.py` provides sufficient behavioral coverage.

### 10. Needs human judgment ‚Äî `bridge-runner.test.ts` internal access patterns

**File:** `bridge-app/src/main/__tests__/bridge-runner.test.ts`  
**Tests:** Several tests access private state via `(runner as unknown as Record<...>).pluginBridge`
to call `.emit()` on the internal bridge, then assert on `runner.getStatus()` and
`mockFetch.mock.calls`.

**Classification:** These are integration tests of `BridgeRunner`'s internal state
machine. The access through the type cast is necessary because the bridge events only
come from the internal `PluginBridge`. The assertions on `mockFetch.mock.calls` are
checking at the process boundary (HTTP calls to the backend). This pattern is
acceptable for a stateful component like `BridgeRunner` where behavior can only
be triggered by internal events.

**Recommendation:** Keep. These tests are testing real observable behavior (health
check triggering stop, now-playing POST containing correct track data). The type cast
is an integration test seam, not a change detector.

## Gaps to fill

### Gap 1 ‚Äî Kiosk portal tests not run in CI

`kiosk/wifi-portal/test_portal.py` has ~40 meaningful tests for config parsing,
nmcli output parsing, HTML generation, and input validation. None of these run in
CI (the `ci.yml` has no kiosk job). If `portal.py` breaks, no automated test fails.

**Severity:** Medium. The kiosk is deployed to physical hardware at events.
`portal.py` is pure Python stdlib ‚Äî it's testable without hardware.

**Proposed fix:** Add a `kiosk` CI job that runs `cd kiosk/wifi-portal && python -m pytest test_portal.py -q`.

---

### Gap 2 ‚Äî Bridge-app coverage thresholds too low

`bridge-app` has branch threshold 55%, function threshold 70%. The bridge-app
is DJs' primary installation artifact (shipped as `.exe`, `.dmg`, `.AppImage`).
`BridgeRunner` is extensively tested (~30 tests), but service files
(`auth-service`, `events-service`, `event-health-service`) have only
a few tests each.

**Severity:** Medium. A shipping binary should have higher confidence than 55% branch coverage.

**Proposed fix:** Raise thresholds incrementally ‚Äî target 75% branch, 85% function.
Write missing tests for error paths in `auth-service.ts` (token refresh failure,
network error) and `events-service.ts` (pagination, event not found).

---

### Gap 3 ‚Äî No property-based tests for scoring/normalization logic

Modules with non-trivial arithmetic and many input combinations have only
hand-crafted example tests:
- `server/app/services/recommendation/scorer.py` (BPM/key/genre scoring)
- `server/app/services/track_normalizer.py` (fuzzy matching, scoring)
- `server/app/services/version_filter.py` (fuzzy matching thresholds)
- `bridge/src/deck-state-manager.ts` (state machine transitions)

For these, property-based testing would provide much higher confidence than
adding more examples. Python has `hypothesis` (already possible in the venv),
TypeScript has `fast-check`.

**Severity:** Low‚ÄìMedium. The existing example tests catch obvious regressions;
property tests would surface edge cases (e.g., BPM near 0, Unicode track names,
empty genre lists) that examples miss.

**Proposed fix:** Ask before introducing `hypothesis` (not yet in `pyproject.toml`)
and `fast-check` (not in `package.json`). If approved, add ~5 property tests each
for `scorer.py` and `track_normalizer.py` to start.

### Gap 4 ‚Äî Missing regression test for 6-hour deduplication window removal

**Commit:** `8ba58a2 fix: remove 6-hour time window from request deduplication (#288)`
(most recent commit). There is no test in `test_dedup.py` that pins the
*time-window behavior* ‚Äî specifically, that a duplicate submitted 7+ hours later
is still detected.

**Severity:** Medium. This was a named fix; without a test it can silently re-enter.

**Proposed fix:** In `test_dedup.py`, add:
```python
def test_deduplication_has_no_time_window(db, test_event):
    """Regression: #288 ‚Äî dedup must apply across the full event lifetime,
    not just within a 6-hour window. A request submitted >6h after the
    original must still be detected as a duplicate."""
    key = compute_dedupe_key("The Killers", "Mr. Brightside")
    db.add(SongRequest(
        event_id=test_event.id,
        song_title="Mr. Brightside",
        artist="The Killers",
        source="spotify",
        status=RequestStatus.NEW.value,
        dedupe_key=key,
        created_at=utcnow() - timedelta(hours=8),
    ))
    db.commit()
    result = find_duplicate(db, test_event.id, "The Killers", "Mr. Brightside")
    assert result is not None, (
        "Regression #288: duplicate detection must not have a 6-hour window"
    )
```
